# Repository Guidelines

## プロジェクト構成とモジュール整理
- `index.html` にページ構造とメタデータ、外部リソースの参照をまとめます。
- `style.css` はレイアウト共通ルール→コンポーネントの順で記述し、可読性を意識します。
- `main.js` ではイベント処理とロジックを管理し、機能追加時は関数単位で整理してください。
- 画像やフォントを追加する際は `assets/`、追加スクリプトは `scripts/` など用途別ディレクトリに配置し、`index.html` のパスを更新します。

## ビルド・テスト・開発コマンド
- `npx serve .` - ルートディレクトリをホストしてブラウザで動作確認します。
- `npx eslint main.js` - ESLint を導入後に実行し、構文・スタイルの逸脱を検出します。
- `npm run format` - フォーマッタ設定を `package.json` に追加した後に実行し、コミット前の整形を統一します。

## コーディングスタイルと命名規則
- HTML/CSS/JS は 2 スペースインデント、タブは使用しません。
- セマンティックなタグ（`main`, `section` 等）と小文字 ID、意味が伝わるクラス名（例: `.counter-display`）を使います。
- CSS はモバイルファーストで記述し、共有トークン（色・余白）を冒頭にまとめます。
- JavaScript は ES6+（`const`/`let`、アロー関数、単一責務のヘルパー）を基本としてください。

## テスト指針
- 手動テスト: ローカルサーバーを起動→ボタンを複数回クリック→リロードして 0 に戻ることを確認します。
- 自動テスト導入時は `tests/` に `main.test.js` 形式で配置し、DOM 更新と操作イベントを網羅します。
- 重要なユーザー操作（クリック、キーボード操作）を 100% カバーしてからマージします。

## コミットと PR ガイドライン
- Conventional Commits を推奨（例: `feat: add reset button`, `fix: prevent negative counts`）。
- PR には背景、実装概要、手動テスト結果（GIF または箇条書き）を記載し、関連 Issue をリンクします。
- レビュー依頼前に lint/format/test を通し、差分のスクリーンショットやデモ URL があれば添付してください。
